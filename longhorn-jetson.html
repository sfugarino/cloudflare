Kubernetes Storage on ARM64

What Longhorn Is, and How to Run It on Jetson Devices

Author: Samuel Fugarino

Longhorn is a cloud-native, distributed block storage system for Kubernetes. It gives you persistent volumes with replication, snapshots, backups, and an easy web UI. On Jetson (ARM64), Longhorn can work well for edge clusters when iSCSI and kernel prerequisites are configured correctly.

What Longhorn Does

Longhorn creates and manages Kubernetes Persistent Volumes by running a lightweight storage engine on each node. Each volume can be replicated across nodes for higher availability.

Core capabilities

  • Dynamic volume provisioning using Kubernetes StorageClass.
  • Replica-based high availability across multiple nodes.
  • Volume snapshots and recurring backup jobs.
  • Easy operational visibility through the Longhorn UI.
  • Good fit for edge and homelab clusters where local disks are common.

Why Jetson users pick it

  • No separate SAN required.
  • Runs directly in your Kubernetes cluster.
  • Works with mixed workloads (AI services, databases, telemetry pipelines).

Jetson Prerequisites

Ubuntu 22.04 ARM64 K3s or K8s open-iscsi
  1. All Jetson nodes must have working Kubernetes networking.
  2. The iSCSI stack must be installed and active.
  3. The iscsi_tcp kernel module must load successfully.
  4. Each node needs a writable disk path for Longhorn data (for example /var/lib/longhorn).
On some Jetson kernels, iscsi_tcp is not built by default. Build and load it first.

Step-by-Step Setup (Jetson)

1. Install host dependencies on every Jetson node

sudo apt update
sudo apt install -y open-iscsi nfs-common
sudo systemctl enable --now iscsid

2. Confirm iSCSI kernel support

sudo modprobe libiscsi
sudo modprobe iscsi_tcp
lsmod | grep -E "iscsi_tcp|libiscsi"

If iscsi_tcp is missing, compile and copy modules as described in your kernel module build notes.

iSCSI TCP Kernel Module Setup Guide

3. Install Longhorn via Helm

kubectl create namespace longhorn-system
helm repo add longhorn https://charts.longhorn.io
helm repo update
helm install longhorn longhorn/longhorn \
  --namespace longhorn-system \
  --set defaultSettings.defaultDataPath="/var/lib/longhorn"

4. Verify all Longhorn components are ready

kubectl -n longhorn-system get pods
kubectl -n longhorn-system get daemonset
kubectl get sc

5. Access the UI and test a volume

kubectl -n longhorn-system port-forward service/longhorn-frontend 8080:80

Open http://127.0.0.1:8080, then create a test PVC and attach it to a pod.

Recommended Jetson Settings

  • Use at least 3 nodes for real replica-based availability.
  • Set Longhorn replica count to 2 or 3 depending on node count and storage budget.
  • Use SSD or NVMe where possible for better latency.
  • Keep node clocks in sync (NTP) for stable distributed operations.
  • Reserve CPU and memory for Longhorn system pods on smaller Jetson boards.

Common Problems on Jetson

  • Pods stuck in Init: Usually missing iSCSI tooling or module.
  • Volume attach fails: Check iscsid status and node kernel logs.
  • Replica scheduling failures: Not enough free disk or too few schedulable nodes.
  • Performance is low: Reduce replica count for dev workloads and move data path to faster storage.

Useful checks

systemctl status iscsid
kubectl -n longhorn-system logs deploy/longhorn-driver-deployer
kubectl -n longhorn-system logs -l app=longhorn-manager --tail=200
Practical note: Verify your Longhorn version supports your ARM64/Jetson environment before production rollout.
